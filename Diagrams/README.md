# Pipelined CPU – Structural Netlist Visualization

## Overview

This directory is dedicated exclusively to the structural visualization of a pipelined CPU. It contains the Verilog RTL, the synthesized unoptimized structural netlist, and the resulting SVG schematic.

The objective of this directory is to:
* Preserve structural hierarchy during synthesis for visual inspection
* Visualize the internal datapath and control flow
* Inspect wide pipeline registers visually (e.g., 96-bit stage registers)
* Enable architectural debugging by tracing exposed internal signals within the generated schematic

## Included Files

* `pipeline.v` — The synthesizable Verilog implementation of the pipelined CPU.
* `pipeline_detailed.json` — The unoptimized structural netlist generated by Yosys.
* `pipeline_detailed.svg` — The visual schematic of the datapath and control logic.

## Generating the Detailed Structural Netlist

We intentionally generate an unoptimized structural netlist to preserve:
* Pipeline registers
* Control signals
* Intermediate wires
* Module hierarchy
* Stage boundaries

### Step 1 – Generate JSON Netlist Using Yosys
Launch Yosys and run the following commands:
```bash
read_verilog pipeline.v
hierarchy -check -top pipeline
proc
memory
fsm
write_json pipeline_detailed.json
exit
```

> **Important:** Do **NOT** run optimization commands like `opt`, `flatten`, `techmap`, or `abc`. These would collapse the hierarchy, remove structural clarity, and ruin the visualization.

### Step 2 – Generate the SVG Schematic
Run the following command in your terminal:
```bash
netlistsvg pipeline_detailed.json -o pipeline_detailed.svg
```

This will generate or overwrite `pipeline_detailed.svg`, which you can then open in any modern web browser or vector graphics viewer.

## Why an Unoptimized Netlist is Used

The primary goal of this directory is architectural visibility, not logic compression or physical implementation.

**An unoptimized netlist:**
* Preserves explicit pipeline registers (e.g., wide 96-bit stage registers).
* Keeps control and datapath signals distinctly visible in the SVG.
* Prevents intermediate wires from being optimized away.
* Maintains a direct, 1:1 traceability from the RTL to the visual structural representation.

**If optimized, the visualization would:**
* Collapse combinational logic into unrecognizable blocks.
* Remove "unused" internal signals critical for human understanding.
* Merge or eliminate registers, obscuring pipeline stage separation.

For debugging and educational inspection, structural transparency is the highest priority.

---

## Exposed Debug Signals in the Schematic

Internal debug signals are intentionally exposed as module outputs in the RTL. When viewing the SVG, this ensures:
* Pipeline stage values (IF/ID, ID/EX, EX/MEM, MEM/WB) remain distinct and visible.
* Control signal propagation can be easily traced across the datapath.
* Hazard handling logic and forwarding paths can be visually verified.
* Write-back paths can be cleanly inspected.

Without explicitly exposing these signals, the synthesis tool might hide or remove them, making the resulting schematic much harder to read and debug.
